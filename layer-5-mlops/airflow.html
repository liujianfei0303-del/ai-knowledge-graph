Airflow

核心要点

Apache Airflow 是一个开源的工作流编排平台，由 Airbnb 于 2014 年创立、2016 年捐赠给 Apache 基金会，以其基于 Python 的 DAG（有向无环图）编程模型成为数据工程和 MLOps 领域的事实标准调度工具。核心概念：DAG（Workflow 的图结构定义，使用纯 Python 代码声明任务依赖关系而非 XML/JSON/YAML）、Operator（任务的原子执行单元，内置 BashOperator、PythonOperator、DockerOperator、KubernetesPodOperator 等数百种社区贡献的 Operator）、Task（DAG 中的单个 Operator 实例，通过 >> 或 set_downstream 定义依赖顺序）、Scheduler（持续监控 DAG 文件的变更，根据 start_date 和 schedule_interval/cron 表达式触发 Task 执行）、Executor（Task 的运行机制，支持 SequentialExecutor（单进程调试）、LocalExecutor（多进程并行）、CeleryExecutor（分布式跨机器）、KubernetesExecutor（每个 Task 启动独立 Pod））。Airflow 的 DAG 编程模型将工作流代码化，天然支持版本控制（DAG 文件存储在 Git 仓库中）、CI/CD 集成（通过 Git Sync 或 Image 部署自动同步 DAG 变更）、以及任务级别的重试和告警（retries、retry_delay、email_on_failure）。在 ML 场景中 Airflow 是数据管道（ETL/ELT）、特征工程流水线、模型定时重训练和推理批处理调度的核心编排工具，与 MLflow、Kubeflow 等 MLOps 组件互补。

主要内容

DAG 编程模型与调度机制

Airflow 的 DAG 定义采用 Python 上下文管理器语法（with DAG(...) as dag:），每个 DAG 包含多个 Task 及它们之间的依赖关系。Task 之间的数据传递通过 XCom（跨任务通信机制）实现——上游 Task 调用 task_instance.xcom_push(key, value) 写入数据，下游 Task 通过 task_instance.xcom_pull(task_ids, key) 读取，XCom 默认存储在 Airflow 的元数据库（MySQL/PostgreSQL）中。Scheduler 的执行策略基于 start_date 和 schedule_interval 生成 DAG Run，每个 DAG Run 对应一个时间窗口的完整执行。Airflow 2.x 引入的 @dag 和 @task 装饰器（TaskFlow API）简化了 DAG 定义，通过函数返回值自动解析 XCom 数据传递，不再需要手动 xcom_push/pull。时序依赖策略包括：depends_on_past=False 允许历史回溯重跑（backfill，通过 airflow dags backfill 命令），catchup=True/False 控制是否补跑 start_date 到当前时间之间的所有未执行的 DAG Run。Airflow 的 SLA 机制允许对指定 Task 设置完成时间约束，超出时触发告警通知。

Executor 架构与资源管理

Airflow 的 Executor 层抽象了 Task 的运行环境。LocalExecutor 在 Scheduler 进程中通过多进程池并行执行 Task，适用于单机开发和测试场景。CeleryExecutor 基于 Celery 分布式任务队列将 Task 分发到 Worker 集群执行，通过 Broker（Redis/RabbitMQ）传递任务消息、Backend（Redis/DB）存储任务状态，Worker 可配置 concurrency 和 prefetch_multiplier 控制并行度。KubernetesExecutor 为每个 Task 创建一个独立的 Kubernetes Pod，执行完毕自动销毁，天然隔离每个 Task 的依赖环境——特别适合 ML 场景中不同 Task 使用不同 Python 依赖和 GPU 资源配置的情形。Executor 的选择影响 Task 的资源隔离粒度：CeleryExecutor 的 Worker 进程共享主机资源，KubernetesExecutor 提供 Pod 级别的资源隔离。此外，Airflow 的 Pool（资源池）机制限制特定 Task 的并发度（如限制同时运行的 GPU 任务数量不超过可用 GPU 数），Queue 机制将特定 Task 路由到指定标签的 Worker（如 queue="gpu" 只调度到 GPU Worker 节点）。

MLOps 集成与 ML Pipeline 实践

在 ML 工作流中 Airflow 的典型管线拓扑为：数据提取（S3/Hive/MySQL → S3 Parquet）→ 数据验证（Great Expectations / Deequ）→ 特征工程（Spark Transformations）→ 模型训练（KubernetesPodOperator 启动 PyTorchJob）→ 模型评估 → 模型注册（MLflow Registry）→ 模型部署（KServe InferenceService 更新）。Airflow 通过 DockerOperator 运行容器化 ML 任务，通过 KubernetesPodOperator 执行需要 GPU 的训练 Pod，通过 PythonOperator 调用 MLflow API 记录实验结果。Airflow 与 MLflow 的典型集成模式是：Airflow 负责 DAG 编排和数据管道，MLflow 负责实验追踪和模型注册，两者互补而非竞争。Airflow 2.x 的 Dataset 感知调度（Data-aware Scheduling）允许 DAG 基于数据集的更新事件触发（而非仅基于时间），更加适配事件驱动的 ML Pipeline 场景。在大规模 MLOps 部署中 Airflow 的传感器（Sensor）机制（如 S3KeySensor、ExternalTaskSensor、TimeDeltaSensor）是实现跨系统编排的关键工具——当上游数据或任务尚未就绪时，Sensor 以可配置的 poke_interval 定期检查条件，超时后触发 timeout 告警或任务失败。

Airflow 是一个基于 Python DAG 编程模型的开源工作流编排平台，通过有向无环图定义任务依赖关系，广泛用于数据管道、定时任务和 ML Pipeline 的调度编排。
